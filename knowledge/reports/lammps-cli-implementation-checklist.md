# LAMMPS 专用 CLI 实施清单

本清单用于记录当前从“嵌入式 LAMMPS agent 模式”推进到“可交付的 LAMMPS 专用 CLI”还需要完成的工作。

## 一、必须做

### 1. 工作流闭环

- 验证 `lammps-coordinator` 能真实调用：
  - `lammps-case-librarian`
  - `lammps-input-writer`
  - `lammps-reviewer`
  - `lammps-analyst`
- 固化 `WF-01` 到 `WF-05` 的推进规则
- 让 reviewer gate 成为默认流程，而不只是 prompt 约束

### 2. 输入生成可靠性

- 给 `lammps-input-writer` 增加“生成后自动复审”闭环
- 建立 5 到 10 个标准输入模板族
- 对高风险场景增加更强约束：
  - ReaxFF
  - MEAM
  - 多元体系
  - 应变计算
  - restart / data 续算

### 3. 专用 CLI 入口

- 确定最终命令名
- 确定默认主 agent
- 确定默认启动方式：
  - 交互式
  - 参数式
  - 两者并存

### 4. 可重复测试

- 建立固定测试集
- 至少覆盖：
  - 案例检索
  - 输入脚本生成
  - reviewer 审查
  - 日志分析
  - 经验写回

## 二、建议做

### 1. 项目状态管理

- 为每个 LAMMPS 项目定义统一目录结构
- 保存：
  - 当前阶段
  - 关键参数
  - 已审查结果
  - 待解决问题

### 2. 大日志分析优化

- 固化超大 `log.lammps` 的分块分析模板
- 自动抓取：
  - 开头初始化段
  - 报错段
  - 热力学统计段
  - 结尾总结段

### 3. 模板库升级

- 从案例库提炼标准模板，而不是长期直接引用原始案例
- 模板建议分组：
  - 拉伸
  - 剪切
  - 熔化凝固
  - 沉积
  - 反应力场
  - 续算
  - 分析

### 4. 知识库收口与自动学习收口（暂缓）

- 继续清理 `confirmed` / `pending` 规则
- 定义哪些 lesson 可以自动写入
- 定义哪些 lesson 必须先询问用户确认
- 去除重复规则，统一措辞
- 增加 pending -> confirmed 的升级流程
- 增加 lesson 去重与合并规则
- 增加二次审核机制
- 该项暂不手动集中处理，后续应通过“定时启动”的知识整理功能执行

### 5. 定时启动的知识整理功能

- 设计定时任务入口
- 定期扫描新增 lessons、pending 条目、重复规则
- 输出待确认候选、可自动提升候选、冲突候选
- 将整理结果写入专门的维护区，而不是直接改写主知识库
- 支持用户二次确认后再正式合并

### 5. 中文用户文档

- 写一份给最终使用者的命令说明
- 说明每个 agent 的用途、调用方式、适用场景

## 三、以后再做

### 1. HPC 提交 skill

- 通过 `.claude/skills/lammps-hpc-submit/` 提供
- 覆盖：
  - 作业脚本生成
  - 资源估算
  - 提交前检查
- 不作为独立 built-in 功能接入，而是在需要超算工作流时由 coordinator 调用 skill

### 2. 可视化集成

- OVITO 输出模板
- 图表自动生成
- 标准后处理可视化流程

### 3. 结构建模工具链

- Atomsk / OVITO / 后处理脚本联动

### 4. 更强的 CLI 产品化

- 子命令体系，例如：
  - `build`
  - `review`
  - `analyze`
  - `learn`
- 项目初始化命令
- 专属配置文件

## 四、推荐实施顺序

1. 工作流闭环
2. 输入生成可靠性
3. 专用 CLI 入口
4. 固定测试集
5. 项目状态管理
6. 大日志分析优化
7. 定时启动的知识整理功能
8. HPC / 可视化

## 五、当前判断

- 现阶段：知识库底座和 agent 团队已基本成型
- 当前短板：workflow 闭环、默认审查闭环、产品化入口、自动学习收口
- 当前最优先：先把“能稳定完成完整任务链”做扎实
- 知识库收口暂缓，后续通过定时启动的知识整理功能处理
